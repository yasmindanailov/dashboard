Fase 3 · Patrones

DetailPage

El patrón que enmarca la ficha de cualquier entidad. Tres variantes nativas (DD-029): standard (lineal), with-aside (metadata permanente) y workspace-lite (3 columnas para triage). Header limpio sin accent-stripe lateral (DD-030) — la firma Aelium la dan avatar with-status, badge tipográfico, eyebrow opcional, tabular-nums. Tabs vía componente DS (DD-028).

Variante · standard

01

El default. Detalle lineal: el usuario consume top-to-bottom. Las acciones globales viven en el header. Sin metadata densa que requiera estar visible siempre. Caso producto: /cliente/billing/[id] — factura individual.

Cuándo usar cada variante

04
Variante Cuándo Cuándo NO
standard Detalle lineal · acciones globales en header · sin metadata densa permanente Hay metadata estructurada que el usuario necesita siempre visible
with-aside Productividad · acciones rápidas repetidas · agente/admin operativos Cliente final navegando (excepción: services con estado de salud)
workspace-lite Triage · 3 planos: cola → foco → contexto · agente/admin Cliente final · workspace puro de chats (usar Workspace propio)

Anti-patrones

05
  • ❌ Header con StatsCards "para resaltar el ID y el total". El número va inline en h1 con tabular-nums. StatsCards solo en Overview.
  • ❌ Border-left brand en el header card. Reservado a navegación. DD-030 lo prohíbe en recuadros.
  • ❌ Doble breadcrumb (uno arriba + otro dentro del header). Solo el de arriba.
  • ❌ Tabs hardcoded en el wrapper. Usar siempre el componente DS Tabs (DD-028).
  • ❌ With-aside con metadata genérica de relleno. Si no aporta, no va.
  • ❌ Workspace-lite "porque queda pro". Densidad alta exige justificación operativa real.
